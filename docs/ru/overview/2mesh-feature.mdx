Service Mesh — это платформа PaaS на основе Istio, предоставляющая облачные решения для приложений и микросервисов.

Платформа предлагает возможность развертывания и обновления инфраструктуры в один клик, визуализированное управление сервисами, эффективное управление производительностью приложений и высокую доступность и производительность для управления шлюзами сервисов. Она предоставляет комплексное, стабильное, надежное и открытое решение для микросервисов для предприятий, проходящих цифровую трансформацию, помогая повысить эффективность управления сервисами и сократить затраты на обслуживание микросервисной инфраструктуры, минимизируя зависимость разработчиков от фреймворка, позволяя предприятиям сосредоточиться на развитии бизнеса и постоянно улучшать свою конкурентоспособность.

## Полное управление жизненным циклом Service Mesh

Управление сервисной сеткой становится особенно сложным в многокластерных средах. Полное управление жизненным циклом сервисной сетки включает не только развертывание, обновление и удаление сервисной сетки, но и мониторинг ее рабочего состояния в реальном времени, позволяя операционному персоналу обнаруживать аномалии и быстро устранять возникшие проблемы, тем самым обеспечивая стабильную поддержку бизнес-операций.

| Особенность                                          | Описание                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| ---------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Сервисная сетка с несколькими основными кластерами** | Функция управления сервисной сеткой с многоосновной кластерной архитектурой визуализирует процесс развертывания компонентов платформы управления микросервисами.                                                                                                                                                                                                                                                                                                   |
| **Визуальное развертывание и управление сервисной сеткой** | - Администраторы могут легко развернуть все компоненты платформы управления микросервисами, включая Istio, для каждого кластера Kubernetes, требующего возможностей управления сервисами, используя визуальные формы для настройки ключевых параметров. Внешние компоненты также могут быть интегрированы по мере необходимости для обеспечения контейнеризированного управления микросервисами.<br />- Администраторы могут настраивать ресурсную конфигурацию компонентов сетки в зависимости от реального объема бизнеса. |
| **Мониторинг сервисной сетки**                      | Функция мониторинга сервисной сетки включает несколько встроенных панелей мониторинга, которые отображают данные мониторинга управляющей плоскости и плоскости данных с различных точек зрения и гранулярности, упрощая администраторам запросы или просмотр данных в реальном времени.                                                                                                                                                                                        |

Интегрированный с Flagger, это обеспечивает автоматизированный опыт выпуска сервисов, поддерживая множество стратегий развертывания, таких как Canary, A/B Testing и Blue/Green развертывания.

| Особенность                    | Описание                                                                                                                                                                                                                                                   |
| ------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Конфигурация выпуска**       | Определите конфигурации выпуска сервисов, чтобы предоставить параметры выполнения для будущих изменений, позволяя автоматизированное развертывание и откат подов с канарейками.                                                                           |
| **Стратегия маршрутизации трафика** | Предлагает гибкие стратегии маршрутизации трафика. Сервисы могут автоматически маршрутизировать трафик на основе шаблонов конфигурации выпуска или настраивать правила маршрутизации трафика для конкретных задач выпуска.                                |
| **Управление релизами**        | Поддерживает как ручные, так и автоматизированные варианты выпуска. Пользователи могут делегировать решения о релизах автоматизированным системам или выбрать ручной выпуск. Процесс выпуска допускает ручное вмешательство и возможность откатить задачи в любое время. |
| **Наблюдаемость**              | Обеспечивает сравнение метрик и отслеживание трассировок между основной версией и версиями канарейки, позволяя проводить анализ качества и локализацию неисправностей для канареечных релизов.                                                           |

## Управление надежностью сервисных соединений

Сервисы внутри сети взаимодействуют удаленно, используя протоколы, такие как HTTP/HTTP2 и gRPC. Когда ответ сервиса медленный или недоступен, это может вызвать эффект "каскадного сбоя", занимая больше ресурсов от предшествующих сервисов и потенциально приводя к сбою системы.

Чтобы избежать этого, необходимо создать защитные механизмы и симуляции сбоев для вызовов сервисов, чтобы гарантировать надежность сервисных соединений и повысить общую стабильность микросервисной системы.

| Особенность                          | Описание                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Трафиковые стратегии**            | Платформа предоставляет единый обзор управления сервисами, позволяя пользователям настраивать стратегии управления, такие как балансировка нагрузки, управление пулами соединений и разрыв цепей через визуальные формы.<br />- **Стратегия балансировки нагрузки**: Автоматически распределяет трафик среди сервисных подов на основе заданной стратегии балансировки нагрузки.<br />- **Стратегия разрыва цепи**: Обнаруживает ненормальные экземпляры сервисов, гладко обрабатывает исключения, автоматически изолирует неисправные экземпляры и снижает влияние на downstream-сервисы.<br />- **Стратегия настроек пула соединений**: Отклоняет некоторые запросы, настраивая соответствующие параметры, чтобы предотвратить сбои системы из-за чрезмерного трафика.   |
| **Правила маршрутизации и стратегии** | Поддерживает динамическую настройку стратегий восстановления от сбоев и инъекций сбоев, включая инъекцию ошибок, задержку возврата, повтор таймаута, переписывание запросов и зеркалирование трафика во время выполнения сервисов. Эти стратегии помогают обеспечить надежную работу сервисов и предотвратить локальные неисправности от каскадирования на другие сервисы.                                                                                                                                                                                                                                                          |

## Управление безопасностью сервисов

В сценарии управления микросервисами вызовы сервисов переходят от локальных к вызовам интерфейсов сетевого протокола, что влечёт за собой риски безопасности.

| Особенность                     | Описание                                                                                                                         |
| ------------------------------- | --------------------------------------------------------------------------------------------------------------------------------- |
| **Стратегия безопасности**      | Обеспечивает шифрование трафика между сервисами. Устанавливая стратегии безопасности для сервисов, трафик может быть зашифрован с использованием mTLS. |
| **Чёрные и белые списки**      | Настраивая чёрные или белые списки для сервисов, вы можете контролировать разрешения на доступ между сервисами внутри одной сети.  |

## Визуализированные операции и обслуживание сервисов

Когда происходит сбой сервиса, обычный процесс устранения неполадок включает: отслеживание аномального трафика > определение места неисправности > сужение области > анализ конкретной причины. Основываясь на этом процессе, платформа предоставляет полный путь устранения неполадок сервисов, помогая вам быстро определить подозрительные места на основе аномального трафика и решить проблемы быстро, чтобы минимизировать влияние на бизнес.

| Особенность                               | Описание                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ----------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Мониторинг трафика и мониторинг JVM** | Предоставляет многомерные панели данных мониторинга трафика для клиентских и серверных сервисов с гранулярностью до уровня API;<br />- Мониторинг JVM поддерживает сбор и отображение данных мониторинга для различных индикаторов, таких как CPU/память, потоки и классы. Это позволяет просматривать данные одного пода или сравнивать данные двух подов, чтобы помочь разработчикам быстро локализовать индикаторные элементы и временные точки, где возникли конкретные проблемы.                                                                                                                                                             |
| **Оповещения**                           | - Предоставляет функции оповещения на основе общих индикаторов мониторинга, поддерживая сбор индикаторов, управление стратегиями, диагностику срабатывания оповещений и уведомления об оповещениях. В сочетании с мониторингом трафика и мониторингом JVM, это направлено на обеспечение полного **мониторинга-оповещения-уведомления** интеллектуального опыта эксплуатации.<br />- Предоставляет шаблоны конфигурации стратегий оповещения, поддерживая конфигурацию одной стратегии оповещения или быстрое создание стратегий оповещения с помощью шаблонов.<br />- Позволяет просмотреть информацию о ресурсах, находящихся под оповещением, через панель **реального времени оповещений**, фильтруя данные по уровню оповещения и просматривая детали оповещения, чтобы помочь пользователям понять масштаб проблемы и быстро локализовать коренную причину. |
| **Логи в реальном времени**              | Обеспечивает панель логов в реальном времени на уровне сервиса, позволяя вам быстро локализовать и устранить проблемы с помощью журналов (включая журналы доступа), когда возникает аномалия сервиса.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| **Быстрое устранение неполадок**        | Когда происходит сбой сервиса, вы можете комбинировать богатые визуальные данные (топология сервисов, трассировка) и данные мониторинга и операций (мониторинг, оповещения, логи) для отслеживания и локализации аномального трафика и быстрого решения проблем.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |

## Восстановление сервисов после сбоев

В сценариях с несколькими регионами и центрами обработки данных, когда сервисный под в центре обработки данных выходит из строя и изолируется с помощью разрыва цепи, платформа автоматически распределяет

трафик клиентов на сервисные поды в других центрах обработки данных, чтобы обеспечить стабильное предоставление услуг.

| Особенность                           | Описание                                                                                                                                                                                                                        |
| ------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Маршрутизация по близости**        | Все сервисные вызовы по умолчанию используют маршрутизацию по близости во всех центрах обработки данных. Когда узлы цепочек вызовов длинные, это эффективно обеспечивает приоритет обработки трафика восстановления перед другими сервисными конечными точками того же кластера. |
| **Настройка приоритета восстановления** | Позволяет настраивать региональные нагрузки для индивидуализации приоритета нагрузки трафика восстановления по нескольким кластерам на основе региона, в котором находятся кластеры.                                                                      |
