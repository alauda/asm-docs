以下是本文件中出现的专有术语、术语和缩写词的定义。

| 术语                     | 描述                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 集群                     | Kubernetes 集群管理着运行容器所需的基础设施资源，并与多个节点、负载均衡器和私有网络等各种资源相关联。<br />单点集群只有一个控制节点，而高可用集群至少有 3 个控制节点。                                                                                                                                                                                                                                                                                         |
| 节点                     | 集群中的节点分为两种类型：控制节点（Master）和计算节点（Node）。<br />控制节点负责运行 kube-apiserver、kube-scheduler、kube-controller-manager、etcd、容器网络和平台的一些管理组件。<br />计算节点是 Kubernetes 集群中承载工作负载的节点，可以是虚拟或物理机器。计算节点负责实际的 Pod 调度和与管理节点的通信。 |
| 项目                     | 平台的项目（租户）可以灵活地划分为独立且隔离的资源空间，每个项目都有独立的项目环境，可以代表企业内部的不同子公司、部门或项目团队。通过项目管理，可以轻松实现项目团队之间的资源隔离和租户内的配额管理。                                                                                                                |
| 命名空间                 | Kubernetes 命名空间是平台上一种较小的资源空间，且与项目中的其他命名空间隔离，它也是用户实现作业生产的工作区。多个命名空间可以在一个项目下创建，总资源配额不可超过项目配额。通过将资源配额细分为更小的命名空间，命名空间下容器的规模（CPU、内存）也受到限制，从而有效提高资源利用率。  |
| 镜像                     | 容器镜像是包装容器应用程序的标准格式。在部署容器化应用程序时，可以指定镜像，这些镜像可以来自 Docker Hub、公司的 DevOps 平台镜像库或用户的私有镜像库。镜像 ID 可以通过镜像库的 URI 和镜像标签（默认是 latest）唯一确认。                                                                                                                               |
| 容器                     | 基于镜像创建的在主机上运行的进程。                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| Pod                      | Pod 是 Kubernetes 中部署应用程序或服务的最小基本单元。一个 Pod 封装了多个应用容器（或仅一个容器）、存储资源、独立网络 IP 和控制容器运行方式的策略选项。                                                                                                                                                                                                                                                               |
| 工作负载                 | 构成平台上应用程序的组件，统称为可以组合以提供服务或独立运行的程序，基于镜像创建。                                                                                                                                                                                                                                                                                                                                                                                                              |
| 副本控制器              | 副本控制器（RC）确保 Kubernetes 集群中始终有指定数量的 Pod 副本运行。通过监控运行中的 Pod，RC 确保集群中有指定数量的 Pod 副本。指定的数量可以有多个或 1 个。如果少于指定数量，RC 将开始运行新的 Pod 副本。如果超过指定数量，RC 将终止多余的 Pod 副本。                                                                             |
| 副本集                   | 副本集（RS）是 RC 的升级版，唯一的区别是支持选择器。RS 支持更多类型的匹配模式。副本集对象通常不单独使用，而是作为部署的理想状态参数。                                                                                                                                                                                                                                                                                                      |
| 部署                     | 部署为副本集（replication controllers）和 Pod（容器组）提供声明式更新。通过描述副本（Pod 的数量）和容器的期望状态，部署控制器将 Pod 和副本集的实际状态更改为期望状态。                                                                                                                                                                                                                                                             |
| 配置映射                 | Kubernetes 配置映射使用键值对来存储配置数据，可以存储单个属性或配置文件。通过使用配置实现容器化应用程序的配置管理，将配置与镜像内容分离，维护容器化应用程序的可移植性。                                                                                                                                                                                                                                                    |
| 密钥                     | Kubernetes 密钥用于在 Kubernetes 集群中存储敏感信息或配置，例如用户密码、OAuth 令牌、SSH 私钥、访问镜像库的身份验证信息等。建议首先使用密钥来存储机密字典。                                                                                                                                                                                                                                                    |
| 服务                     | Kubernetes 服务定义了一组 Pods 的逻辑集合，并支持为集群内的计算组件设置访问策略。它相当于集群中的内部服务，并为位于集群内的其他计算组件或访问者提供统一个访问入口，实现计算组件的内部发现功能。                                                                                                                                                                                                                           |
| 自定义资源定义           | 在 Kubernetes API 中，资源是用于存储某种类型的 API 对象集合的端点。CRD 对象在集群中定义了一种新的、唯一的对象类型，并允许 Kubernetes API 服务器处理该对象的整个生命周期。Kubernetes 支持用户通过 CustomResourceDefinition API 自定义 Kubernetes 扩展资源，确保新的资源可以快速注册和使用。                                           |
| 域                       | 管理员可以使用域名管理功能集中管理用于该平台的企业网络域名资源，并通过将域名绑定到项目在项目之间分配和管理域名资源。                                                                                                                                                                                                                                                                                                                             |
| 角色                     | 角色是一组操作权限。平台上资源的操作权限包括创建、查看、更新和删除。平台通过角色对不同资源的操作权限进行分类和组合。角色可以对一种或多种类型的资源具有一个或多个操作权限。通过将特定角色分配给用户，用户可以快速开启或限制指定资源的操作权限。                                                                                                                     |
| 事件                     | 平台与 Kubernetes 事件集成，记录 Kubernetes 资源的重要状态变化以及各种运行时状态变化事件。通过事件，可以分析集群、应用程序、任务等特定资源中的异常情况的具体原因。                                                                                                                                                                                                                                                          |
| 审计                     | 平台与 Kubernetes 审计集成，提供与安全相关的按时间顺序的操作记录，包括时间、来源、操作结果、发起操作的用户、操作的资源及请求/响应的详细信息等。通过审计，平台审计员可以清楚地了解 Kubernetes 集群的变化。                                                                                                                                                                                                        |
| 日志                     | 平台与 Kubernetes 日志集成，可以快速收集 Kubernetes 集群中的容器日志，包括容器的标准输出和容器内的文本文件。同时，支持收集 Kubelet、Docker 和 Kubernetes 容器化组件的日志。                                                                                                                                                                                                           |
| 令牌                     | 系统向用户发放的令牌，携带用户的身份和权限等信息。当用户调用 API 时，令牌会被添加到请求消息头中，通过身份验证可以获得调用 API 操作资源的权限。                                                                                                                                                                                                                                                                 |
| 微服务                   | 微服务是一种用于开发软件的架构和组织方法，由小型、独立的服务组成。平台上的服务指的是内部路由及其相关的计算组件。                                                                                                                                                                                                                                                                                                                                                                            |
| 微服务                   | 微服务是一种软件架构和组织方法，由小型、独立的服务组成。平台上的服务指的是内部路由及其相关的计算组件。                                                                                                                                                                                                                                                                                                                                                                            |
| 服务网格                 | 服务网格是用于微服务应用的可配置基础架构层。它使得每个服务实例之间的通信更加流畅、可靠和快速。服务网格提供了一系列功能，如服务发现、负载均衡、加密、身份认证、授权、支持断路器模式等功能。                                                                                                                                                                     |
| 外部服务                 | 外部服务是相对概念，对于服务网格内的服务而言，所有服务外部的都是外部服务。                                                                                                                                                                                                                                                                                                                                                                               |
| 微服务网关               | 作为微服务架构的入口和出口，服务网关（Istio 网关）可以为微服务治理平台提供路由转发、API 管理、权限验证过滤等能力。服务网关包括入口网关和出口网关。                                                                                                                                                                                                                                        |
| 入口网关                 | 入口网关是 Istio 中重要的资源对象之一，定义所有通过服务网格的入站流量的入口点。它用于在网格边缘管理入站流量并进行路由。                                                                                                                                                                                                                                                                                                                                 |
| 出口网关                 | 出口网关是 Istio 中重要的资源对象之一，定义服务网格的出流量的出口点。已注册在网格中的外部服务可以绑定到出口网关，使得访问流量从出口网关发送到外部。                                                                                                                                                                                                                                         |
| 跟踪                     | 跟踪是单个请求调用的链过程，TraceID 是该请求调用的唯一标识。当调用关系复杂时，多个调用链具有相同的 TraceID。                                                                                                                                                                                                                                                                                                                                |
| 边车                     | 边车模式是一种分布式架构的设计模式，通过将应用功能拆分为单独的进程，分离控制和逻辑。在微服务系统中，集成在应用程序中的服务功能被移到边车中，边车可以作为独立进程在应用程序中部署，提供服务发现、注册、服务调用、应用身份验证、速率限制等功能。                                                                      |
| 服务拓扑                 | 服务拓扑是指平台上服务调用关系的可视化拓扑图。通过查看服务拓扑，可以了解平台上服务调用及其在指定时间范围内的性能。                                                                                                                                                                                                                                                                                                                  |
| API 组                   | API 分组是服务网关管理项目中用于管理服务和服务 API 的一个维度。它聚合了一组具有共同场景的访问资源，并统一管理和监控它们。                                                                                                                                                                                                                                                                                                                |
| Envoy                    | Envoy 是为大型现代 SOA（面向服务的架构）设计的 L7 代理和通信总线。它是 Istio 的核心组件之一，与服务以边车的方式运行，以拦截并转发服务的流量。它提供流量路由和流量控制等能力。[了解更多](https://www.envoyproxy.io/docs/envoy/latest/intro/what_is_envoy)                                                                                                                     |
| EnvoyFilter              | EnvoyFilter 提供了一种机制，可以自定义 Istio Pilot 生成的 Envoy 配置。EnvoyFilter 可以修改特定字段的值，添加特殊过滤器或新增完全不同的监听器、集群等。该功能应谨慎使用，因为错误的配置可能会导致整个服务网格崩溃。<br />平台上的 EnvoyFilter 资源是项目资源，可以在同一项目内多个服务和命名空间中共享。          |
